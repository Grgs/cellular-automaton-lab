# Tiling Known Deviations

This file tracks known mismatches between the current app implementation and the strongest literature-faithful target we eventually want.

## Current Blocking Deviations

None.

## Current Known Deviations

- `dodecagonal-square-triangle` runs a decorated 3.12.12 Archimedean generator (each regular dodecagonal supercell decomposed into six unit squares plus twelve unit equilateral triangles, plus two bridging triangles per supercell). It tiles the plane exactly, has no depth cap, and uses no vendored data, but it is not the canonical Schlottmann quasi-periodic square-triangle tiling: the supercell layout is locally 6-fold symmetric and the global tiling is periodic at the supercell scale.
- `pinwheel` now renders as a clean two-root substitution patch (the previous chaotic look came from a vertex-ordering bug in the second root that made `_map_local` a non-similarity transform — fixed by listing the second root's vertices in the canonical `(small-angle, right-angle, large-angle)` order so the subdivision rule maps to similar children). The patch keeps both roots as `left` chirality at depth 0; the substitution rule (3 right + 2 left children per left parent) introduces full chirality variety from depth 1 onward. A literature comparison on June 11, 2026 against the Bielefeld pinwheel patch found that the same bug class persisted one level deeper: three of the five subdivision children were declared with non-canonical vertex order, so the second application of the subdivision sheared the grandchildren (equal areas, balanced chirality, but only 20/50 canonical `1:2:sqrt(5)` triangles at depth 2). A same-day promotion was reverted, the three child tuples were reordered to canonical `(small-angle, right-angle, large-angle)` form, and every tile is now congruent to the prototile at every depth, with the depth-1 chirality split matching the published 3-right + 2-left rule per left parent. The reference verifier now enforces per-tile congruence (`expected_triangle_side_ratios`) for `pinwheel` and `pinwheel-2-1`, so an angle-mismatched affine subdivision can no longer pass automated gates. Because the similarity-correct subdivision is genuinely not edge-to-edge, the float conversion of its exact rational coordinates leaves hairline T-junction gaps that split Shapely's polygon-union surface at depth 3 (a `1e-9` buffer reunifies it); `pinwheel` therefore carries the same `polygon_surface_check=False` waiver as `pinwheel-2-1`, while graph connectivity, overlap, and edge-multiplicity checks all stay strict. Manual visual review accepted the corrected congruent patch on June 12, 2026 (congruent-tile field matching the Bielefeld reference texture, all twelve angular sectors occupied, zero representative render-space overlaps), and `pinwheel` now ships in the main `Aperiodic` group.
- `taylor-socolar` is the half-hex substitution from Bielefeld `/substitution/half-hex/` (attributed there to Grünbaum & Shephard 1987 / Frettlöh 2002) with the **chirality bit** of the full Taylor-Socolar labelling layered on top. Per Baake-Gähler-Grimm 2012 (arXiv:1210.3967), the bare half-hex substitution is a topological factor of the Socolar-Taylor LI class (Lemma 1 / Corollary 3): TS adds a 14-prototile labelling (7 hex colours A-G × 2 chiralities) on the same geometric substitution. This implementation carries the chirality bit by flipping chirality on the one substitution child that is geometrically reflected (the `paired` child, rotated by π); the other three children inherit the parent's chirality. Both roots seed with chirality `left`, so the central pair forms one full chiral hexagon and the depth-d L/R split balances toward 50/50 (`2/0, 6/2, 20/12, 72/56` at depths `0..3`). Cells are emitted as two distinct kinds (`taylor-half-hex-left`, `taylor-half-hex-right`) with `chirality_token` set, so palette, renderer, and CA rules can distinguish them. What is **not** yet computed: the 7-colour A-G labelling that the full Taylor substitution carries alongside chirality, which determines the exact per-hex R1 stripe direction and R2 flag positions from Socolar & Taylor 2011 (arXiv:1009.1419) Fig. 1, Fig. 4, and Fig. 9. The published Taylor-Socolar matching rules (R1: black stripes must be continuous across shared edges; R2: vertex flags at the endpoints of a shared edge must point in the same direction) require those marking positions. Closing this gap requires transcribing the 14-state substitution table from Baake-Gähler-Grimm Fig. 7 / Socolar-Taylor Fig. 9 and extending the canvas adapter to overlay the R1 black stripe + R2 vertex flags per hex.
- `pinwheel-2-1` is the two-prototile pinwheel variant from the Bielefeld Tilings Encyclopedia (https://tilings.math.uni-bielefeld.de/substitution/pinwheel-2-1/). The seed pairs two 1:4:√17 right triangles into a 4:1 rectangle in canonical (small-angle, right-angle, large-angle) vertex order; each parent subdivides into one small child at the right-angle corner (scale 1/√17) plus four large children filling the rest (scale 2/√17), so the depth-d total is 2 · 5^d. All construction points (foot of altitude at `(64/17, 16/17)` and the three midpoints) are rational, so the generator runs in exact `Fraction` arithmetic and grandchildren are similar to the parent at every depth. The two prototile sizes are emitted as distinct cell kinds (`pinwheel-2-1-small-triangle`, `pinwheel-2-1-large-triangle`) so renderer, palette, and rules can distinguish them. The substitution inherently produces T-junctions and point-only cell contacts (some cells touch their neighbors at single vertices rather than along edges), so the polygon-union surface-component check in `topology_validation.recommended_validation_options` is skipped for this family — the cell-adjacency graph is still required to be connected, and overlap and edge-multiplicity checks still apply. Status is `exact_affine`. Every tile is congruence-verified against the `1:4:sqrt(17)` prototile at each depth (machine-enforced by `expected_triangle_side_ratios`), and a manual visual review on June 13, 2026 accepted the rendered patch against the published Bielefeld pinwheel-2-1 patch (matching elongated-fan texture with both prototile sizes interspersed, zero representative render-space overlaps via the `pinwheel-2-1-depth-3` render-review profile), so the family now ships in the main `Aperiodic` group.
- `stein-14-pentagonal` is the Stein 1985 monohedral convex pentagonal tiling, completely determined by `2a = 2c = d = e`, `A = 90°`, `2B + C = 360°`, `C + E = 180°` with `sin(B) = (√57 − 3) / 8`. The pentagon shape and 6-tile p2 primitive unit are exact (ported from Stein's original Java construction algorithm). The tiling is **genuinely non-edge-to-edge**: pentagon vertices sit on the midpoints of other pentagons' edges, and the vertex coordinates are irrational. The midpoint relationship cannot be enforced bit-exactly in float64 (~5×10⁻⁷ drift between the stored vertex and the geometric midpoint), so shapely sees polygon intersections of order 1×10⁻⁵ even though the tiling is geometrically valid. To accommodate this, `topology_validation` maintains an `_NON_EDGE_TO_EDGE_IRRATIONAL_GEOMETRIES` allowlist (currently `{"stein-14-pentagonal"}`) that opts the geometry into looser polygon-overlap (`1e-3`) and surface-hole (`1e-2`) tolerances. All other geometries keep the strict `1e-7` tolerance. Future tilings combining T-junctions with irrational coordinates (Stein-15, some Reinhardt/Rice pentagons) can opt into the same allowlist. The neighbor builder's T-junction detection pass (`_attach_neighbors` second pass) handles the half-edge adjacency automatically for any non-edge-to-edge tiling, so no descriptor flag is required for that.

## Known Limits That Are Not Currently Treated As Failures

- `penrose-p3-rhombs` and `penrose-p3-rhombs-vertex` are built by the de Bruijn pentagrid construction in `backend/simulation/penrose.py` rather than by iterating the canonical Robinson half-tile substitution from a seed. The pentagrid is mathematically equivalent to the canonical Penrose rhomb substitution and produces valid thick / thin rhombs with correct matching at every depth, so the families are flagged as `canonical_patch` rather than `known_deviation`. The trade-off is that the depth-to-cell-count sequence (5/10/24/66 at depths 0..3) is governed by the bounding-box crop at half-extent `0.85 * phi^d` rather than by the substitution eigenvalue; the analogous half-tile rewrite (acutes at short edges into thin rhombs, obtuses at long edges into thick rhombs) was attempted but does not converge to a clean rhomb tiling at depth >= 2 from any rhomb-star or sun seed examined so far. See `docs/PENROSE_CANONICAL_SUBSTITUTION_PLAN.md` for the original investigation.
- `penrose-p1-pentagon-diamond` is the distributed Penrose P1 variant, built by the de Bruijn pentagrid construction (de Bruijn 1981; Pattern Collider by Aatish Bhatia) followed by a P3 → P1 vertex-merge pass, implemented in `backend/simulation/aperiodic_penrose_multigrid.py`. Stage 1 runs the pentagrid with non-uniform offsets `(0.3, 0.4, 0.5, 0.6, 0.7)` to get a regular Penrose P3 rhomb tiling distributed across the patch without a central singularity. Stage 2 walks every rhomb-vertex and identifies the four canonical Penrose vertex configurations — sun (5 thick rhombs at 72° apex), star (10 thin rhombs at 36° apex), and two 3-rhomb boat configurations (1 thin + 2 thick at 144°/108°/108°, or 2 thin + 1 thick at 144°/144°/72°) — collapsing each into the corresponding P1 prototile cell. Sun-merges emit `p1-pentagon-cluster` (10-vertex decagonal cell at the canonical P1 pentagon position, distinct from unmerged thick rhombs); star-merges emit `p1-star`; boat-merges emit `p1-boat`. Unmerged thick rhombs remain as `p1-pentagon` cells (the rhomb-region MLD representative of the pentagonal P1 prototile) and unmerged thin rhombs as `p1-diamond` cells (geometrically exact P1 diamonds). All five P1 cell kinds are distributed throughout the patch, with depth-to-cell-count sequence 42 / 107 / 272 / 723 / ... governed by the `1.6 * φ^d` patch radius. Cells are intrinsically gap-free, edge-matched, and connected at every depth.
- `heptagonal-7-fold` ships the de Bruijn **heptagrid rhombus tiling** (catalog label "Heptagonal 7-fold (rhombs)"), built by the generalized-dual (multigrid) construction in `backend/simulation/aperiodic_heptagonal_7_fold.py` over seven line families spaced `2*pi/7` apart (the shared `aperiodic_penrose_multigrid` builder with `symmetry=7` and seven generic offsets). Because 7 is odd there is no antiparallel-family degeneracy, so all seven families are used directly. The dual is the 7-fold rhomb tiling with prototiles {thin rhomb (acute `pi/7 ~ 25.7 deg`), medium rhomb (`2*pi/7 ~ 51.4 deg`), wide rhomb (`3*pi/7 ~ 77.1 deg`)}; it is the heptagonal analogue of the Penrose (5-fold) and Socolar (12-fold) multigrid rhomb tilings. Status is `canonical_patch`: like those families the patch is deterministic, gap-free, edge-matched, and validator-clean, but the depth-to-cell-count sequence (62/139/317/707 at depths 0..3) is governed by the bounding-box crop half-extent `1.0 * 1.5^d` rather than by a substitution eigenvalue, and matching rules are not decorated onto the tiles. This is **not** the Goodman-Strauss 7-fold tiling, which is a different, marked-prototile *substitution* construction; only its rhombus-grid relative is shipped here. The shared multigrid sampler's default sector nudge (`1e-3`) is too coarse for the narrow 7-fold sectors and produced a few malformed quadrilaterals, so the heptagrid builder passes a finer `1e-5` nudge, after which every emitted cell is a true unit rhombus.
- `penrose-p1-pentagon-boat-star` is the centered singular-pentagrid Penrose P1 manifestation. It uses the all-zero de Bruijn pentagrid crop so the patch includes the iconic central star and surrounding boat ring from depth 0, and it emits the full `pentagon` / `diamond` / `boat` / `star` vocabulary. Status is `canonical_patch`: the patch is deterministic, gap-free, edge-matched, and validator-clean, but it is still a canonical multigrid patch rather than the literal decorated six-state substitution implementation.
- Periodic mixed families are still verified on finite open-boundary canonical samples rather than larger-sample or quotient-surface proofs. The reference layer now supports per-family periodic sample sizes, but the current audit kept the shipped catalog on `3x3`.
- Periodic dual-family invariants now cover the unambiguous reciprocal pairs plus candidate-class signature checks for the currently ambiguous catalog groups, but they are still finite-sample descriptor checks rather than full dual-construction proofs.
- The reference layer now mixes rooted local-reference anchors with direct canonical patch diffs for `robinson-triangles`, `tuebingen-triangle`, `dodecagonal-square-triangle`, `shield`, and `pinwheel`; it is not yet full canonical patch coverage for every family.
- Several aperiodic families are verified through deterministic low-depth samples, count invariants, adjacency rules, and metadata presence rather than a full symbolic substitution proof.
- Render-space overlap checks still need a looser tolerance than the backend topology-space overlap check; the current frontend helper uses a `1e-4` positive-area threshold plus multi-precision snapped comparison because exact-path families like `pinwheel` are still noisier in adapter space.
- Pinwheel contiguity now uses exact positive-length segment-overlap neighbors on the exact-affine path because the substitution is not edge-to-edge at every subdivision step.

## When To Add An Entry Here

Add a known deviation when:

- the generator is intentionally approximate
- the literature verifier is knowingly weaker than the intended family model
- a family is temporarily kept on a waiver list
- rendering is accepted as a stopgap even though the mathematical generator is not yet canonical

Do not add an entry just because a family uses a finite sample. Finite samples are part of the app’s model and only count as a deviation if they fail to preserve the intended invariants.
